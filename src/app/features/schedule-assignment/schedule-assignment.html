Problem Statement Take Home Test

Your task is to design a simple UI feature to schedule a server reboot.

Requirements:
  • Users can select a server.
  • Users can schedule a time for the reboot.
Instructions & Context:
  • Assume the application uses Angular and TypeScript.
  • Your focus should be on the UI design and logic only.
  • Assume that all necessary backend services and APIs are available and functional.
  • This exercise reflects the kinds of problems we solve day-to-day, the goal is to see how you structure and implement a feature - clarity, maintainability and logical design are important.
Solution Architechture Developed by Hussain Ali Shahid
Overview:

The project was structured following best practices for a scalable Angular application. The architecture separates concerns into the following modules:

  • layout: Contains consistent components used across the entire application, such as the header, footer, and content wrapper.
  • core: Contains constants, enums, route guards, interceptors, interfaces, services
  • feature: Contains components that correspond to specific application pages or routes.
  • shared: Comprises reusable components, such as forms and dialog boxes, that are shared across multiple feature modules.
Implementation Highlights
  • Initial Project Setup: The foundation was laid by installing the Angular CLI and integrating Angular Material. External styling from Bootstrap CDN was also imported.
  • API Communication: A server.ts service was created to manage communication with the application's API.
  • Data Models: Type safety was ensured by creating interfaces for both API responses (server.interface.ts) and specific request payloads, such as the reboot request (reboot-request.interface.ts).
  • Feature Development: A dedicated page (schedule-assignment) was developed to provide an explanation of the feature's implementation.
Implementation of the Server Reboot Feature
  • Reboot Form (server-form)
    • Server Selection: Users can select a specific server from a list populated by the getAll API response from the server.ts service.
    • Reboot Timing: The user can choose to reboot the server "Now" or "Later" using a flag.
      • Selecting "Later" dynamically enables additional form controls for scheduling.
      • If the "Later" option is chosen, the user can specify the exact date and time for the reboot.
    • Reason for Reboot: A text field is available for users to provide a reason for the reboot.
    • Form Validation: All form controls include robust validation rules (e.g., required, minLength, maxLength) to ensure data integrity before submission.
  • Confirmation Dialog (confirmation-dialog)
    • The dialog content is dynamically generated based on the selected reboot strategy ("Now" or "Later"), providing context-specific information to the user.
  • API Communication and User Feedback
    • Upon successful form submission and user confirmation, the application sends the reboot request to the API via the server.ts service.
    • After receiving the API response, a Snackbar notification is displayed to the user, confirming the status of the reboot request.
